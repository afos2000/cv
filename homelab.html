A 4-node k3s cluster running on bare-metal hardware at home. All nodes are connected via a flat Layer 2 network. The control plane runs on the master node; all three workers handle workloads. External access is exclusively via Cloudflare Tunnel — zero open ports on the home router.

root@andreskube:~# cat /etc/cni/net.d/kube-router.conf

# Network stack

kube-router handles both pod networking (CNI) and NetworkPolicy enforcement. Each pod gets a unique IP from the cluster CIDR. Network policies follow a default-deny model — only explicitly allowed ingress/egress is permitted.

# External access

Cloudflare Tunnel creates an encrypted outbound connection from the cluster to Cloudflare's edge. DNS records with proxy enabled (orange cloud) route traffic through Cloudflare's network, eliminating the need for any open firewall ports. Cloudflare Access adds SSO authentication in front of sensitive services like ArgoCD.

# TLS termination

Cloudflare edge terminates external TLS with auto-managed certificates. Internal traffic uses cert-manager with an internal CA for cluster-internal TLS. Forward mTLS (planned via Linkerd) will encrypt traffic between pods.

01
Push to main — All cluster configuration lives in a single Git repository. Every change starts with a pull request or direct push.
02
ArgoCD detects drift — ArgoCD polls the Git repo (or receives webhooks). When it sees the desired state diverges from the live cluster, it reconciles.
03
Apps of Apps — A root ApplicationSet defines the entire cluster. Each component (Longhorn, cert-manager, etc.) is an ArgoCD Application managed by the parent.
04
Self-healing — If someone makes a manual change via kubectl, ArgoCD reverts it to match the repo within the sync interval.
